# 瑞普数据同步系统 - 需求文档（V1）

## 1. 项目背景与目标
为郑州瑞普生物工程有限公司建设一套可扩展的数据同步与数据查询系统。第一阶段对接氚云 OpenAPI，实现表单数据采集、增量同步、后台管理与对外查询 API；后续可扩展对接 SQLServer 备份文件、金蝶等数据源。

## 2. 范围定义
### 2.1 一期范围（必须交付）
1. 氚云表单同步（全量初始化 + 增量更新）
2. 每个表单独立同步策略（自动/手动、周期可配）
3. 后台管理系统（登录、表单配置、API Key 管理、数据查看、导出）
4. 数据查询 API（API Key 鉴权 + 表单级权限控制）
5. 全链路日志（同步日志、API 查询日志、后台审计日志）
6. 表单中文备注和 API 中文备注管理，并在 AI 查询输出中返回备注信息

### 2.2 二期预留（不在一期开发）
1. SQLServer 备份文件接入
2. 金蝶接入
3. 更复杂的数据编排与跨源数据治理

## 3. 用户角色
1. 系统管理员
- 管理登录账户
- 配置表单同步策略
- 管理 API Key 与权限
- 查看日志与系统状态

2. API 调用方（AI/业务系统）
- 通过 API Key 查询已采集数据
- 按授权范围访问指定表单

## 4. 功能需求
### 4.1 后台登录与权限
1. 提供后台登录页面
2. 账号密码登录（密码哈希存储）
3. 登录会话管理（过期、退出）
4. 所有后台关键操作写入审计日志

### 4.2 表单同步管理
1. 可新增/编辑同步表单配置
- 表单 SchemaCode
- 表单显示名称
- 表单中文备注
- 同步方式（AUTO/MANUAL）
- 同步周期（按分钟配置）
- 同步模式（INCREMENTAL/FULL）

2. 支持手动触发单表单同步
3. 自动调度按表单独立周期运行
4. 后台可查看每个表单最近同步状态、时间、条数、错误信息

### 4.3 数据落库与动态建表
1. 每个表单对应一张业务数据表
2. 表单字段按氚云 1:1 存储
3. 额外保留 `raw_json` 完整字段（供 API/AI 查询）
4. 一期假设表单结构稳定不变；系统仍保留扩展点支持未来字段变更处理
5. 不做删除同步（源端删除不在目标端删除）

### 4.4 增量同步
1. 采用 `ModifiedTime + ObjectId` 作为增量游标
2. 同步逻辑支持幂等 upsert
3. 单次任务失败可重试
4. 任务执行前后写入同步日志

### 4.5 数据查看与导出
1. 后台可按表单浏览已同步数据
2. 支持筛选（字段条件）
3. 支持导出（CSV/Excel）
4. 后台默认展示业务字段，不展示 `raw_json`（可在详情页或调试模式查看）

### 4.6 API Key 与查询 API
1. 后台生成 API Key
- 名称
- 中文备注
- 过期时间
- 可访问表单集合（表单级权限）

2. API 查询能力
- 标准 REST 查询
- 条件过滤、分页
- 基础关联查询（跨已授权表单）
- 返回结果包含表单备注、字段备注，便于 AI 理解数据语义

3. API 调用日志
- key 标识
- 调用时间
- 请求参数摘要
- 响应条数
- 耗时
- IP
- 成功/失败

## 5. 非功能需求
1. 稳定性：核心任务失败可重试，任务不重复并发执行
2. 安全性：密钥哈希存储、过期控制、后台鉴权、操作审计
3. 可维护性：模块化设计（Connector 插件化）
4. 可扩展性：保留多数据源接入接口
5. 可观测性：结构化日志 + 数据库业务日志

## 6. 规模与容量（当前预估）
1. 表单数量：约 50
2. 单表总量：约 5,000
3. 日增量：10-100/表单
4. 结论：单体应用 + PostgreSQL 可稳定支撑一期

## 7. 约束与假设
1. 运行环境：Ubuntu
2. 数据库：PostgreSQL
3. 部署方式：单体服务（不前后端分离）
4. 当前表单结构短期稳定
5. 删除数据不回写目标库（只增/改）

## 8. 验收标准（一期）
1. 可配置并同步至少 50 个氚云表单
2. 增量同步准确，重复执行无脏数据
3. 后台可完成登录、配置、触发同步、查看数据、导出
4. API Key 可控到表单级，并支持过期
5. API 查询返回包含备注信息，AI 可直接理解表与字段含义
6. 同步/API/后台操作均有可检索日志
